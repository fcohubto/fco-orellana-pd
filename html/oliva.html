OLIVA OS — Framework de Ejecución con IA

Un sistema operativo diseñado en tres capas: biblioteca de skills, protocolo de ejecución y sistema operativo del estudio. No es un conjunto de herramientas — es una arquitectura de trabajo.

Lo diseñé siendo usuario y arquitecto al mismo tiempo. Eso obliga a un nivel de honestidad diferente — si el sistema no funciona en la práctica, lo cambia quien lo usa.

El desafío

El problema no es falta de IA — es falta de sistema

El problema

Usar IA sin protocolo produce resultados inconsistentes. Cada sesión reinicia el contexto, cada output tiene distinta calidad. El problema no es la herramienta — es la ausencia de un modelo de ejecución que estructure cómo y cuándo activar cada capacidad.

El objetivo

Diseñar una arquitectura que defina cómo operar con IA: qué profundidad aplicar por tipo de tarea, qué skills activar en cada contexto y cuándo un resultado es suficientemente bueno para avanzar — sin depender del estado de ánimo de la sesión.

Hipótesis de partida: La diferencia entre usar IA con impacto real y usarla como autocomplete no está en el acceso a las herramientas — está en si existe un sistema de ejecución detrás de cada interacción.
Metodología de diseño

Diseñar la arquitectura como si fuera un producto

Cada capa tiene una responsabilidad específica, criterios de entrada y salida, y puede evaluarse de forma independiente. El mismo rigor que aplico a productos de cliente lo apliqué a la arquitectura del propio sistema.

Capa 01: Arquitectura

Tres capas, tres responsabilidades

OLIVA OS no es una colección de prompts ni una herramienta. Es una arquitectura donde cada capa tiene su propia lógica — y puede evolucionar sin romper las demás.

Capa 01

Skills — La biblioteca base

Repositorio de 23 skills organizados por tipo de tarea. No son prompts sueltos — son unidades de trabajo con input esperado, proceso y output definido. La materia prima del sistema.

Capa 02

Oliva Framework — El protocolo de ejecución

Los tres modelos de ejecución (Rápido, Controlado, Profundo) que determinan qué skills activar y en qué secuencia según la complejidad de la tarea. Es el núcleo cognitivo del sistema.

Capa 03

Olivacraft — Sistema operativo del estudio

El flujo completo de pre-venta a cierre: 11 etapas, 9 documentos, criterios de avance por escrito. La capa que el cliente experimenta — y donde el sistema toca la realidad.

Por qué tres capas y no un sistema unificado

Cada capa puede evolucionar de forma independiente. Si cambio cómo opero con un cliente (Capa 03), no necesito tocar el framework de ejecución (Capa 02). La separación de responsabilidades no es teórica — es lo que hace sostenible el sistema en el tiempo.

Cómo se conectan las capas

Capa 01 provee las unidades de trabajo. Capa 02 define el protocolo para combinarlas. Capa 03 aplica ese protocolo a los proyectos reales del estudio. El flujo siempre va de lo abstracto a lo concreto — de skills a operación.

Capa 02: Oliva Framework

Tres modelos para tres niveles de profundidad cognitiva

No toda tarea requiere la misma densidad de proceso. El Oliva Framework define cuántos prompts, qué secuencia y qué criterios aplicar según la naturaleza de lo que se está ejecutando.

Rápido 6 prompts por sesión

Para tareas tácticas con contexto claro. Output en una sesión, sin iteración profunda. Ideal para ajustes menores o entregables dentro de proyectos activos.

Controlado 11 prompts por sesión

Para tareas con complejidad media. Incluye contexto, ejecución y revisión estructurada. Es el modelo más usado en la práctica diaria del estudio.

Profundo 16 prompts por sesión

Para proyectos estratégicos o de alta ambigüedad. Diagnóstico, hipótesis, ejecución, validación y síntesis. Requiere bloques de tiempo sin interrupción.

Cómo se elige el modelo

No es una preferencia — es una lectura del problema. Si la tarea tiene contexto claro y output predecible: Rápido. Si requiere decisiones con criterio propio: Controlado. Si la ambigüedad es el punto de partida: Profundo.

Qué determina cada modelo

La cantidad de prompts define el ciclo completo: cuántos pasos de diagnóstico, cuántas iteraciones de output y cuántas instancias de revisión antes de cerrar la sesión.

Principio de diseño

Profundidad proporcional a ambigüedad

Un sistema que trata todas las tareas igual produce resultados igualmente mediocres. Aplicar 16 prompts a un ajuste de copy es ineficiente. Aplicar 6 prompts a una decisión estratégica es riesgoso. El modelo existe para calibrar esa diferencia.

Capa 03: Olivacraft

El sistema operativo del estudio en práctica

La Capa 03 traduce la arquitectura a operaciones concretas: cada interacción con un cliente tiene un protocolo, cada documento tiene una versión canónica y cada etapa tiene criterios de entrada y salida.

Qué existe en el sistema hoy

Formulario de contacto calificado, email de bienvenida, propuesta estándar, contrato base, brief estructurado, emails de avance por etapa, checklist de entrega, email de cierre y protocolo de seguimiento a 30 días.

Qué no tiene el sistema

Casos de borde sin resolver. Cada excepción que encontré en la práctica se volvió una regla escrita — o una razón para no aceptar ese tipo de proyecto. El sistema crece con experiencia real, no con teoría.

11 etapas · Estructura del flujo

Pre-venta: Contacto → Calificación → Propuesta → Contrato

Ejecución: Brief → Arquitectura → Diseño → Revisión → Aprobación

Cierre: Entrega → Handoff → Seguimiento 30 días

Criterio de diseño

El cliente percibe orden antes de ver una pantalla

Antes de recibir un diseño, el cliente ya vivió un proceso claro. Eso comunica solvencia profesional antes que portafolio.

En uso

El sistema opera en cada proyecto activo

No es una documentación — es la forma en que ejecuto cada sesión de trabajo. Cada skill tiene un uso registrado en proyectos reales.

Validado en práctica

Cada modelo fue calibrado con proyectos reales

Los 6, 11 y 16 prompts no son números teóricos. Son el resultado de medir cuándo una sesión produce un output suficientemente bueno para avanzar.

Diferenciador

Ejecución con IA que no depende del humor de la sesión

La consistencia no viene de la disciplina — viene de tener un sistema que define el proceso antes de que empiece la sesión.

Vivo

El sistema evoluciona con cada proyecto

Cada caso raro se vuelve una regla nueva. Cada skill que no funciona se reemplaza. El sistema crece con experiencia real, no con iteraciones teóricas.

Conclusión

Un sistema es útil cuando lo usas — no cuando lo describes

OLIVA OS no existe para demostrar que uso IA con sofisticación. Existe porque sin él la ejecución es inconsistente. Un framework que funciona en la práctica vale más que uno que suena bien en papel.
